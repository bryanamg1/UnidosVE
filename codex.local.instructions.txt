UNIDOSVE - LOCAL AGENT INSTRUCTIONS

Objective
- Build the UnidosVE frontend as a React + Vite solidarity platform for Venezuela.
- Support donation flows, collection centers, active needs and coordination with a future Flask backend.
- Keep the UI layer isolated from data origin so mock data can be replaced with real backend services later.

Active Frontend Scope
- Frontend root: venezuela-sos-front/
- Current phase control is mandatory. Do not advance to the next phase without explicit user approval.
- Phase 0 only covers setup, local instructions, folder structure, constants and backend-gap documentation.

Frontend Stack
- React 19 + Vite
- React Router DOM for navigation
- Material UI for atomic and form components
- CSS Modules for layout and feature-specific styling
- React Context for mock authentication and role state
- react-leaflet + leaflet with OpenStreetMap for the interactive map

Backend Status
- Current backend is partial and lives separately under venezuela-sos/backend.
- Verified endpoints currently available:
  - GET /api/health
  - GET /api/mensajes
  - POST /api/mensajes
  - GET /api/usuarios-salvo
  - POST /api/usuarios-salvo
- The backend does not yet provide real endpoints for collection centers, needs, donations, login, register or role management.
- Do not invent backend endpoints for missing features.

Mock-First Rule
- Use local mocks initially for centers, needs, donations, auth and roles.
- Keep contracts stable through services, adapters, hooks and contexts.
- Components must not depend on whether data comes from mocks or HTTP.
- Future migration must replace service implementations, not visual components.

Architecture Rules
- Use feature-based architecture under src/features.
- Keep presentational components small and focused.
- Do not call fetch or axios directly from visual components.
- Keep logic in hooks, services, adapters, contexts or utils.
- Centralize texts, labels, routes, storage keys, statuses, map config and HTTP config in src/constants.
- Keep mocks inside their matching feature folder.
- Adapters must normalize mock and backend entities into stable frontend shapes.

Required Frontend Layers
- constants: shared config, texts, routes, statuses, storage keys
- context: auth state and future shared app contexts
- routes: public, protected and role-based routing
- services: API client and data-source aware services
- features: landing, auth, centers, needs, donations, map, dashboard
- hooks and utils: shared logic with no visual coupling

Design Rules
- Visual direction: dark theme, strong blue primary, green success, amber warning, red critical
- Use Material UI for buttons, inputs, cards, chips, tabs, dialogs, alerts and loaders
- Use CSS Modules for layout, hero sections, dashboard composition, map wrappers and responsive behavior
- Avoid hardcoded copy inside JSX

Map Rules
- Use Leaflet + OpenStreetMap only during the mock-first phase
- No Google Maps for now
- Map must support center markers, user geolocation when permitted and graceful handling of denied/unavailable geolocation

Roles
- donor
  - Can view landing, donation feed, map, nearby centers, published needs and own donation status
  - Can initiate donations
  - Cannot publish needs or manage centers
- center
  - Can register/login in mock auth
  - Can register and edit its center profile
  - Can publish and update needs
  - Can review associated donations and confirm received donations
  - Must not see donor-only forms
- public
  - Can access landing and public entry actions
  - Cannot manage needs or confirm donations

Permission Rules
- Protect private screens with auth-aware route guards
- Enforce role access through dedicated route wrappers, not ad-hoc JSX conditions scattered across views
- Keep permissions declarative in constants whenever possible

Git Flow
- Start each new task from develop when the repository is initialized
- Branch naming convention: bryan/<type>/<short-name>
- Example branches:
  - bryan/feat/project-foundation
  - bryan/feat/landing-page
  - bryan/feat/mock-auth
- At the end of each phase prepare a commit, then merge into develop
- Merge into main only after explicit user approval
- If git is not initialized in the active frontend folder, document the intended flow and wait for repository setup

Phase Control
- Never move to the next phase automatically
- Valid user approvals include:
  - vamos con la fase 0
  - vamos con la fase 1
  - hagamos el cierre de fase 1
  - vamos con el merge hacia develop

Token Optimization
- Keep explanations compact
- Avoid dumping full files unless necessary
- Summarize changes by outcome and touched areas

Change Documentation Rule
- Every change must leave a short note covering:
  - what changed
  - why it changed
  - whether it affects future backend migration
